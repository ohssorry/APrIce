> 이 문서는 **"누가 무엇을 하는가"가 아니라 "누가 무엇을 하지 않는가"** 를 정의한다.
> [`ROADMAP.md`](./ROADMAP.md) 가 *일정과 할 일*이라면, 이 문서는 *권한과 책임의 경계*다.

> **읽는 법**: 자기 파트(§2 은지 / §3 가영 / §4 예은)만 읽어도 완결되게 썼다.
> 단, **§5 공통 규칙**과 **§6 이 문서가 깨지는 상황**은 세 명 모두 읽는다.

## 1. 경계의 대전제

### 1.1 제품 원칙 (경계를 나누는 기준)

| 원칙 | 경계에 미치는 영향 |
|---|---|
| **로컬 CLI 실행이 메인 사용 경로**, GitHub Action은 안전망 | CLI가 먼저 완성되고, Action은 CLI를 감싸기만 한다. Action 이슈로 CLI 설계를 바꾸지 않는다 |
| **웹은 정적 HTML 리포트 수준** (풀스택 대시보드 아님) | 서버·DB·API 엔드포인트·로그인은 **누구의 영역도 아니다.** 만들자는 제안 자체가 스코프 위반 |
| **세 영역의 교차점은 결과 JSON 스키마** | 이 스키마는 **은지가 초안을 잡고 팀이 합의**한다. 합의 전엔 아무도 의존 코드를 쓰지 않는다 |

### 1.2 디렉토리 소유권 (물리적 경계)

```
packages/
├── cli/          👤 은지 단독
├── action/       👤 은지 단독
├── core/         👤 가영 단독
├── report/       👤 예은 단독
├── comment/      👤 예은 단독
├── pricing/      👤 예은 소유 · 스키마는 가영과 공동
schemas/          🤝 3인 공동 — 변경 시 전원 합의 필수
fixtures/         🤝 3인 공동 — 변경 시 전원 합의 필수
.github/workflows/ 👤 은지 단독
README.md         👤 예은 단독
```

**철칙**: 남의 디렉토리 파일은 **한 글자도 고치지 않는다.** 고쳐야 하면 소유자에게 요청한다.
예외: 오타 수정도 예외가 아니다. (남의 파일을 고치는 순간 머지 충돌 책임이 모호해진다)

### 1.3 3인 공동 소유 자산

| 자산 | 내용 | 변경 절차 |
|---|---|---|
| `schemas/analysis-result.schema.json` | 결과 JSON 스키마 = **제품의 헌법** | §5.5 절차 준수 |
| `fixtures/sample-result.json` | 위 스키마의 실제 예시 | 스키마와 항상 동기화 |
| `.aprice.yml` 스펙 | 사용자 설정 파일 형식 | 은지 초안 → 3인 합의 |
| `ROADMAP.md` / `ROLES.md` | 계획과 경계 | 합의 후 PR |

## 2. 은지 — 백엔드 / CLI / GitHub 연동

> **한 줄 정의**: 나는 **제품의 골격**을 만든다. 데이터가 흐르는 파이프와, 그 파이프를 GitHub에 꽂는 일.
> 나는 **숫자를 계산하지 않고, 화면을 그리지 않는다.**

### 2.1 내가 반드시 해야 하는 것

#### 기반 (Day 1)
- [ ] **E1** pnpm workspace 모노레포 세팅 — 6개 패키지 디렉토리 생성, `tsconfig.base.json`, ESLint, vitest 설정
- [ ] **E2** `schemas/analysis-result.schema.json` **초안 작성** 후 3인 리뷰 소집

#### CLI 본체 (Day 2~4)
- [ ] **E3** `aprice init` 서브커맨드 — 대화형 프롬프트로 `.aprice.yml` 생성
- [ ] **E4** 설정 로더 함수 `loadConfig(path): AprIceConfig` — YAML 파싱 + 기본값 병합 + 스키마 검증 + 한글 에러 메시지
- [ ] **E5** 파일 수집기 `collectFiles({ mode: 'all' })` — `.gitignore`·`node_modules`·`*.test.*` 제외
- [ ] **E6** 파일 수집기 diff 모드 `collectFiles({ mode: 'diff', base })` — `git diff --name-only base...HEAD` 파싱, 삭제 파일 제외
- [ ] **E7** `aprice scan` 서브커맨드 — 수집기 → `@aprice/core` 호출 → `aprice-result.json` 파일 출력
- [ ] **E8** 터미널 출력 포맷터 — provider별 표 + 색상 + 결론 3줄
- [ ] **E9** exit code 정책 — `0` 통과 / `1` 예산 초과 / `2` 실행 오류

#### GitHub 연동 (Day 4~6) ⭐ 내 핵심 난이도 구간
- [ ] **E10** `action.yml` 작성 + `@vercel/ncc` 로 `dist/index.js` 단일 파일 번들
- [ ] **E11** PR 코멘트 게시 — `@actions/github` 사용, **기존 코멘트 검색 후 업데이트** (신규 생성 X)
- [ ] **E12** Check Run 등록 — 예산 초과 시 `conclusion: 'failure'` 로 머지 차단
- [ ] **E13** 권한·토큰 설정 문서 초안 — `permissions: pull-requests: write` 등, 예은에게 넘길 텍스트
- [ ] **E14** 데모 레포 `aprice-demo` 구축 — 통과 PR 1개 + 초과 PR 1개를 열어둔 상태

### 2.2 내가 절대 건드리면 안 되는 것

| 건드리면 안 되는 것 | 소유자 | 이유 |
|---|---|---|
| `packages/core/` 의 AST 탐지 로직 | 가영 | 탐지 규칙은 오탐률과 직결 — 내가 임의로 손대면 가영의 테스트 기준선이 무너진다 |
| 비용 계산식 / 시나리오 배수(multiplier) 계산 | 가영 | 숫자의 정합성은 한 사람이 책임져야 검증 가능하다. 두 명이 손대면 틀린 숫자의 출처를 못 찾는다 |
| `packages/report/` 의 HTML·CSS | 예은 | 리포트 디자인 일관성은 한 사람이 잡아야 한다. 내가 "급하니까" 넣은 마크업이 예은의 레이아웃을 깬다 |
| **PR 코멘트의 문구·마크다운 구조** | 예은 | 게시 *메커니즘*은 내 것이지만 *내용*은 예은 것. 내가 문구를 바꾸면 예은이 설계한 정보 위계가 무너진다 |
| `pricing/*.yml` 의 가격 숫자 | 예은 | 출처(`source_url`)와 검증 날짜(`checked_at`) 책임이 예은에게 있다. 내가 고치면 출처 추적이 끊긴다 |
| `README.md` | 예은 | 문서 톤과 스크린샷 관리가 예은 몫. E13은 **초안 텍스트만** 넘기고 편집은 맡긴다 |
| 서버·DB·인증 코드 | **아무도 아님** | 스코프 아웃. "Action 결과를 어디 저장해두면 좋지 않을까"는 Day 7 이후 백로그 |

> ⚠️ **내가 가장 넘기 쉬운 선**: `scan` 커맨드를 배선하다 보면 core 내부가 답답해서 직접 고치고 싶어진다.
> **고치지 말고 가영에게 "이 입력에 이 출력이 필요하다"고 인터페이스로 요청한다.**

### 2.6 막혔을 때 물어볼 곳

```
기술적 질문
├─ AST·탐지·비용 계산이 이상하다      → 가영
├─ HTML/CSS·마크다운 렌더링이 깨진다   → 예은
└─ GitHub Actions·npm·TS 빌드         → 내 영역 (2시간 룰 적용, §5.6)

설계/스키마 질문
└─ 결과 JSON·설정 파일 형식            → 가영 + 예은 동시 소집 (혼자 바꾸지 않는다)

팀 전체 결정
└─ §5.7 절차 — 팀 채널에 [결정필요] 태그로 올린다
```

> 💬 **은지에게만 해당하는 주의**: GitHub Action 인증·권한 문제는 이 프로젝트에서 **가장 오래 막힐 확률이 높은 구간**이다.
> Day 5 오후까지 E11이 안 되면 **혼자 더 파지 말고 팀에 올려서** "로컬 CLI 데모 메인 전환" 플랜 B를 발동한다.

## 3. 가영 — AI · 분석 엔진

> **한 줄 정의**: 나는 **제품의 두뇌**를 만든다. 코드를 읽어 "얼마"라는 숫자를 만드는 일.
> 나는 **파일을 직접 찾지 않고(은지가 준다), 숫자를 직접 보여주지 않는다(예은이 그린다).**

### 3.1 내가 반드시 해야 하는 것

#### 설계 (Day 1)
- [ ] **G1** `AnalysisResult` 데이터 모델 정의 — 필드별 타입·단위·의미를 표로 문서화
- [ ] **G2** `fixtures/` 에 테스트용 소스 파일 6종 작성 (provider별 1개), 각 파일에 "탐지돼야 할 호출 N개" 주석 명시

#### 탐지 (Day 2~3)
- [ ] **G3** `parseFile(path): AST` 래퍼 — `@babel/parser` 로 JS/TS/TSX 파싱, 문법 오류 시 throw 대신 `{ skipped, reason }` 반환
- [ ] **G4** 탐지기 v1 — 직접 호출 패턴 (`openai.chat.completions.create(...)` 등) 탐지
- [ ] **G5** 탐지기 v2 — import 별칭 추적 (`import OpenAI from 'openai'` → 변수 바인딩 따라가기)
- [ ] **G6** 호출 단위 추정기 — 루프·`map`·배열 순회 내부 호출에 `multiplier` 부여 + `multiplierReason` 문자열 생성

#### 계산 (Day 4~5)
- [ ] **G7** 시나리오 엔진 — `config.traffic` (DAU × 요청/유저/일) × multiplier → 월 예상 호출 횟수
- [ ] **G8** 비용 계산기 — 가격 YAML × 월 호출 횟수 → `estimatedMonthlyUsd` (소수점 2자리)
- [ ] **G9** 신뢰도 산정 — `high`/`medium`/`low` 3단계, 판정 기준을 코드 주석과 문서에 동일하게 기재
- [ ] **G10** 예산 비교 — 총액 vs `budget.monthlyUsd` → `verdict: pass | warn | fail`
- [ ] **G11** `core` 단위 테스트 — 탐지·계산·시나리오, 커버리지 70%+
- [ ] **G12** 미지원 provider 처리 — 탐지는 하되 `estimatedMonthlyUsd: null` + 사유 표기

### 3.2 내가 절대 건드리면 안 되는 것

| 건드리면 안 되는 것 | 소유자 | 이유 |
|---|---|---|
| `packages/cli/` 의 커맨드·인자 파싱 | 은지 | CLI UX는 은지가 일관되게 잡는다. 내가 옵션을 추가하면 `init`/`scan`/`report` 간 규칙이 어긋난다 |
| `git diff` 실행 / 파일 시스템 탐색 | 은지 | core는 **경로 배열을 받기만** 한다. 내가 파일을 직접 찾으면 core가 git에 의존하게 되어 테스트가 불가능해진다 |
| `packages/action/` 및 워크플로 YAML | 은지 | GitHub 런타임 디버깅은 은지가 컨텍스트를 쥐고 있다 |
| **`pricing/*.yml` 의 가격 숫자와 `source_url`** | 예은 | 나는 가격을 **소비**할 뿐이다. 계산이 안 맞을 때 숫자를 고치면 원인이 데이터인지 로직인지 영영 못 가린다 |
| HTML 리포트의 표·색상·문구 | 예은 | 내가 "이 필드를 보여줘야 하는데" 싶으면 **필드를 결과 JSON에 넣고 예은에게 알린다.** 직접 그리지 않는다 |
| exit code 값 결정 | 은지 | `verdict` 는 내가 만들지만 그걸 프로세스 종료 코드로 번역하는 건 CLI 계층의 책임 |

> ⚠️ **내가 가장 넘기 쉬운 선**: 계산 결과가 이상할 때 `pricing/*.yml` 숫자를 "잠깐만" 고쳐보는 것.
> **고치지 말고 테스트에서 하드코딩 픽스처를 쓴다.** 실제 YAML은 예은에게 수정 요청.

### 3.3 팀이 같이 결정해야 하는 것 (혼자 결정 금지)

| 항목 | 왜 혼자 결정 못 하나 | 데드라인 |
|---|---|---|
| `AnalysisResult` 필드 구조 | 은지가 직렬화하고 예은이 그린다 | **Day 1 저녁** (IP-1) |
| 가격 YAML 스키마 (`unit`, 무료 티어 표현 방식) | 예은이 6종을 이 형식으로 채운다. 나중에 바꾸면 예은 작업이 통째로 재작업 | **Day 1 저녁** (IP-2) |
| `multiplier` 추정 규칙 (루프 = 몇 배로 볼 것인가) | 숫자의 신뢰도 = 제품 신뢰도. 심사위원 질문 1순위 | **Day 3 종료** |
| `confidence` 3단계 판정 기준 | 예은이 리포트에 배지로 노출하고, 발표에서 설명해야 함 | **Day 4 종료** |

## 5. 공통 규칙 (3인 전원)

### 5.1 브랜치 전략

```
main                    ← 항상 빌드되는 상태 유지. 직접 push 금지
├─ feat/cli-scan        ← 은지
├─ feat/core-detector   ← 가영
└─ feat/report-html     ← 예은
```

| 규칙 | 내용 |
|---|---|
| 브랜치 명명 | `feat/` · `fix/` · `docs/` · `chore/` + 영역-내용 (예: `feat/core-scenario`) |
| 수명 | **최대 2일.** 그 이상 살아 있으면 쪼개서 머지한다 |
| 최신화 | 매일 아침 `git pull --rebase origin main` |
| `main` 직접 push | **금지.** 단, Day 1의 초기 세팅(E1)만 예외 |

### 5.2 커밋 메시지 형식

```
<type>(<scope>): <제목 50자 이내, 한글 가능>

[본문 — 왜 이렇게 했는지. 무엇을 했는지는 diff가 말해준다]
```

- **type**: `feat` `fix` `docs` `refactor` `test` `chore`
- **scope**: `cli` `core` `action` `report` `comment` `pricing` `schema`

```bash
# 좋은 예
feat(core): 루프 내부 API 호출에 multiplier 부여
fix(cli): diff 모드에서 삭제된 파일이 스캔 대상에 포함되던 문제 수정
chore(schema): estimatedCallsPerMonth 필드 추가 (팀 합의 2026-07-24)

# 나쁜 예
수정
wip
asdf
```

### 5.4 코드 컨벤션

| 항목 | 규칙 |
|---|---|
| 언어 | TypeScript, `strict: true` |

| 파일명 | `kebab-case.ts` |
| 함수/변수 | `camelCase`, 타입·인터페이스 `PascalCase` |
| export | 패키지 루트 `index.ts` 에서만 public API 노출. 남의 패키지 내부 경로 직접 import 금지 |
| 주석 | **무엇을**이 아니라 **왜**를 쓴다. 특히 비용 계산식에는 근거 주석 필수 |
| 사용자 노출 문자열 | 한글 (에러 메시지, CLI 출력, 리포트) |
| 코드 내 식별자 | 영어 |

### 5.5 ⚠️ 스키마 · 인터페이스 변경 규칙 (가장 중요)

> **결과 JSON 스키마는 제품의 헌법이다.** 여기가 흔들리면 3명이 동시에 멈춘다.

**변경 절차 (예외 없음)**

1. 팀 채널에 `[스키마변경]` 태그로 올린다 — **무엇을 / 왜 / 누가 영향받는지**
2. 나머지 2명이 👍 를 남긴다 (**2명 모두** 필요)
3. `schemas/` 와 `fixtures/sample-result.json` 을 **같은 PR에서 함께** 수정
4. PR 제목에 `chore(schema):` 접두사
5. 머지 후 팀 채널에 완료 공지 + 각자 `pull` 확인

**시점별 허용 범위**

| 시점 | 필드 추가 | 필드 이름 변경 | 필드 삭제 | 타입 변경 |
|---|---|---|---|---|
| Day 1~2 | ⭕ 자유 | ⭕ 자유 | ⭕ 자유 | ⭕ 자유 |
| Day 3~5 | ⭕ 합의 후 | ❌ **금지** | ❌ **금지** | ❌ **금지** |
| Day 6~7 | ⚠️ 데모에 필수인 경우만 | ❌ 금지 | ❌ 금지 | ❌ 금지 |

**공개 함수 시그니처도 동일하게 취급한다.**
`analyze()` · `renderReport()` · `renderComment()` · `loadConfig()` · `collectFiles()`
→ 파라미터·반환 타입 변경 시 **호출하는 사람에게 먼저 알린다.**

**필드를 지우고 싶을 때 (Day 3 이후)**: 지우지 말고 **`deprecated` 주석만 달고 방치**한다. 7일 프로젝트에서 필드 하나 남는 비용 < 3명이 동시에 깨지는 비용.

### 5.6 2시간 룰

> **혼자 2시간 이상 같은 문제에 막히지 않는다.**

1. 2시간 경과 → 팀 채널에 공유 (시도한 것 3가지 + 에러 메시지)
2. 4시간 경과 → 해당 작업을 **일단 mock/하드코딩으로 우회**하고 다음 작업으로 이동
3. 반나절 경과 → §5.7 절차로 올려 **스코프 축소를 논의**

7일 프로젝트에서 하루를 한 사람이 막히는 데 쓰면 전체 일정의 14%가 사라진다.

### 5.7 팀 결정 절차

```
[결정필요] 태그로 팀 채널에 올린다
   ↓
① 결정할 것 (한 줄)
② 선택지 2~3개
③ 내 추천 + 이유
④ 언제까지 결정해야 하는지
   ↓
3명 중 2명 동의 → 결정 확정
   ↓
결정 내용을 ROADMAP.md 또는 ROLES.md에 반영 (PR)
```

- **결정을 미루지 않는다.** 데드라인까지 답이 없으면 **제안자가 결정하고 진행**한다.
- 확정된 결정은 **뒤집지 않는다.** 뒤집으려면 새 `[결정필요]` 를 올린다.

### 5.8 데일리 체크 (매일 밤 10분)

1. 오늘 항목 중 **완료 기준을 충족한 것만** 체크한다 (80% 완료 = 미완료)
2. 내일 시작할 항목의 **input이 도착했는지** 확인 — 안 왔으면 지금 요청
3. 2시간 룰에 걸린 게 있으면 공유
4. `main` 이 빌드되는지 확인

## 6. 이 문서가 깨지는 상황 (역할 경계 재논의 트리거)

> 아래 상황이 **하나라도** 발생하면, 그날 밤 **30분 안에 팀 전원이 모여** 역할 경계를 다시 그린다.
> "어떻게든 되겠지"로 넘기지 않는다. 7일 프로젝트에서 경계가 흐려지면 회복할 시간이 없다.

### 6.1 일정 지연 트리거

| # | 상황 | 즉시 조치 |
|---|---|---|
| T1 | **한 항목이 예상보다 2일 이상 지연** | 해당 항목을 **스코프에서 제거**하거나 다른 팀원에게 이관. 지연자를 돕는 게 아니라 **일을 줄인다** |
| T2 | **IP-1(결과 JSON 스키마)이 Day 1 저녁에 확정되지 않음** | 🚨 최고 위험. Day 2 오전을 **전원 스키마 확정에만** 쓴다. 다른 작업 전면 중단 |
| T3 | **IP-3(파이프라인 관통)이 Day 3 종료 시점에 실패** | 정확도를 포기하고 **하드코딩 값으로라도 파이프를 먼저 뚫는다.** 가영은 Day 4에 보정 |
| T4 | **IP-4(리포트 연결)가 Day 5 종료 시점에 실패** | HTML 리포트를 **마크다운 출력으로 다운그레이드.** PR 코멘트만 사수 |
| T5 | **Day 6 종료 시점에 E2E가 한 번도 성공하지 못함** | Day 7을 **버그 수정 전용**으로 전환. 발표 자료는 현재 상태로 동결 |

### 6.2 경계 침범 트리거

| # | 상황 | 즉시 조치 |
|---|---|---|
| T6 | **남의 디렉토리를 고쳐야만 진행되는 상황이 반복(2회 이상)** | 경계선이 잘못 그어진 것. 해당 모듈의 소유권을 한 사람에게 재배정 |
| T7 | **같은 파일에서 머지 충돌이 3회 이상 발생** | 그 파일은 공동 소유가 되어버린 것. 파일을 쪼개거나 소유자를 명시 |
| T8 | **"이건 누구 일이지?" 질문이 나옴** | 그 자리에서 이 문서에 **한 줄 추가**하고 커밋. 기억에 의존하지 않는다 |
| T9 | **한 사람이 다른 사람 작업을 기다리며 반나절 이상 놀고 있음** | 의존 관계 설계 실패. mock으로 우회하거나 작업 순서를 재배치 |

### 6.3 스코프 변경 트리거

| # | 상황 | 즉시 조치 |
|---|---|---|
| T10 | **"서버 하나 만들면 되지 않나?" 류의 제안이 나옴** | 즉시 기각하고 **Day 7 이후 백로그**에 적는다. 논의 시간 5분 초과 금지 |
| T11 | **탐지 정확도가 Day 4에 80% 미만** | provider를 **6종 → 3종(OpenAI/Anthropic/Twilio)** 으로 축소. 가영·예은 작업 동시 경감 |
| T12 | **GitHub Action 권한 문제가 Day 5 오후까지 미해결** | **로컬 CLI 데모를 발표 메인으로 전환.** Action은 녹화 영상으로 대체. 은지의 남은 시간은 CLI 완성도에 투입 |
| T13 | **npm 패키지명 `aprice` 선점 확인** | 즉시 대체명 결정 (`aprice-cli` 등). README·발표자료·데모 레포 전부 영향 |
| T14 | **팀원 1명이 하루 이상 이탈 (건강·개인사)** | 그 사람의 **Day 6~7 항목부터 잘라낸다.** 핵심 파이프라인(E7·G4·Y6)은 남은 2명이 분담 |

### 6.4 재논의 시 지켜야 할 것

- **사람을 조정하지 말고 일을 조정한다.** "야근해서 따라잡자"는 해법이 아니다
- **핵심 흐름 3개는 마지막까지 지킨다**: `aprice scan` 동작(E7) / 탐지+계산(G4·G8) / 결과를 사람이 읽는 형태(Y5·Y6)
- 재논의 결과는 **반드시 이 문서와 ROADMAP.md에 반영**하고 커밋한다. 구두 합의는 3일 뒤 사라진다

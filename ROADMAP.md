# APrIce — 개발 로드맵 & 역할 분담

> **APrIce** (구 `tollgate`) — *API Price*
> 유료 외부 API 호출을 **정적 분석**으로 탐지해, **PR 단계에서 월 예상 비용**을 계산하고
> 예산을 초과하면 **머지를 차단**하는 CLI + GitHub Action.

---

## 0. 프로젝트 전제 (모든 의사결정의 기준)

| 항목 | 결정 사항 |
|---|---|
| 핵심 유저 | **코드를 짜는 개발자** |
| 메인 사용 경로 | **로컬 CLI 실행** (`npx aprice scan`) — 커밋 전에 개발자가 직접 돌린다 |
| GitHub Action | 로컬에서 놓친 경우를 잡는 **안전망**. 메인 경로가 아니다 |
| 웹 화면 | 풀스택 대시보드 ❌ → **정적 HTML 리포트** ⭕ (비개발자 팀원이 열어보는 용도) |
| 서버 | **없음.** 백엔드 서버·DB·로그인 전부 스코프 아웃 |
| 가격 데이터 | 레포에 커밋된 **정적 YAML 가격 DB**. 런타임 크롤링 없음 |

> ⚠️ **스코프 방어선**: "이거 대시보드로 만들면 멋있지 않을까?" 류의 제안은 Day 7 이후 백로그로 보낸다.
> 7일 안에 **끝까지 동작하는 하나의 흐름**을 만드는 것이 유일한 목표다.

### 기술 스택 (합의됨)

- **언어**: TypeScript (Node.js 20+) — CLI와 GitHub Action이 같은 코드를 공유할 수 있음
- **AST 파싱**: `@babel/parser` + `@babel/traverse` (JS/TS 대상)
- **CLI 프레임워크**: `commander`
- **패키지 배포**: npm (`npx aprice` 로 설치 없이 실행)
- **리포트**: 의존성 없는 단일 HTML 파일 (CSS/JS 인라인)

### 지원 범위 (v1 확정)

- **분석 대상 언어**: JavaScript / TypeScript **만**
- **탐지 대상 API**: OpenAI, Anthropic, Google Maps, Twilio, AWS S3, SendGrid (6종)
- 그 외는 "미지원 provider"로 리포트에 표시만 하고 비용 계산에서 제외

## 1. 역할 분담 총괄

| 팀원 | 책임 영역 | 한 줄 요약 |
|---|---|---|
| **은지** | 백엔드 / CLI / GitHub 연동 | 제품의 **골격**. 데이터가 흐르는 파이프를 만든다 |
| **가영** | AI·분석 엔진 (AST / 비용 계산 / 시나리오) | 제품의 **두뇌**. "얼마"라는 숫자를 만든다 |
| **예은** | 정적 리포트 / PR 코멘트 UX / 가격 DB / 배포 | 제품의 **얼굴**. 숫자를 사람이 읽게 만든다 |

### 소유권 경계 (충돌 방지용)

```
packages/
├── cli/          ← 은지 단독 소유
├── core/         ← 가영 단독 소유 (분석·비용 엔진)
├── action/       ← 은지 단독 소유 (GitHub Action)
├── report/       ← 예은 단독 소유 (HTML 리포트 렌더러)
├── comment/      ← 예은 소유 (PR 코멘트 마크다운 생성기)
└── pricing/      ← 예은 소유 (가격 YAML) / 스키마는 가영과 공동
schemas/          ← 3명 공동 (변경 시 반드시 셋 다 합의)
fixtures/         ← 3명 공동 (테스트용 샘플 데이터)
```

> **규칙**: 남의 디렉토리 파일을 수정해야 하면 직접 고치지 말고 **이슈/메시지로 요청**한다.
> 단, `schemas/`와 `fixtures/`는 3명 합의 후에만 변경한다.

## 2. 은지 — 백엔드 / CLI / GitHub 연동

**책임 영역**: CLI 명령 체계, 설정 파일 로딩, diff 기반 파일 수집, 결과 직렬화, GitHub Action 패키징, PR 코멘트 게시, 예산 초과 시 머지 차단(exit code + check run)

### 할 일 체크리스트

| # | 할 일 | 완료 기준 | 의존 관계 |
|---|---|---|---|
| E1 | 모노레포 초기 세팅 (pnpm workspace, TS 설정, ESLint, vitest) | 루트에서 `pnpm build` / `pnpm test` 가 6개 패키지 전부에서 에러 없이 통과 | 없음 |
| E2 | `schemas/analysis-result.schema.json` 초안 작성 후 3인 리뷰 | 셋 다 승인, `fixtures/sample-result.json` 이 스키마 검증(ajv) 통과 | **가영 G1과 동시 진행 필수** |
| E3 | `aprice init` — 대화형으로 `.aprice.yml` 생성 | 빈 레포에서 실행 시 provider 목록·월 예산·트래픽 프로필이 채워진 `.aprice.yml` 이 생성됨 | 없음 |
| E4 | 설정 로더 (`.aprice.yml` 파싱 + 기본값 병합 + 스키마 검증) | 필드 누락/타입 오류 시 어느 줄이 왜 틀렸는지 한글 에러 메시지 출력 | E3 |
| E5 | 파일 수집기 — 전체 스캔 모드 (`--all`) | `.gitignore` / `node_modules` / 테스트 파일 제외하고 대상 `.js,.ts,.tsx` 경로 배열 반환 | 없음 |
| E6 | 파일 수집기 — **diff 모드** (`--base <ref>`) | `git diff --name-only base...HEAD` 기준으로 변경 파일만 반환, 삭제된 파일은 제외 | E5 |
| E7 | `aprice scan` — core 엔진 호출 → `aprice-result.json` 출력 | 샘플 레포에서 실행 시 스키마를 만족하는 JSON 파일이 생성됨 | E2, **가영 G4** |
| E8 | 터미널 출력 포맷터 (표 + 색상 + 요약 3줄) | 비개발자가 봐도 "무슨 API가 월 얼마"인지 3초 안에 읽히는 출력 | E7 |
| E9 | Exit code 정책 구현 (`0` 통과 / `1` 예산 초과 / `2` 실행 오류) | 예산 초과 픽스처에서 `echo $?` 가 `1`, 정상 픽스처에서 `0` | E7 |
| E10 | GitHub Action 패키징 (`action.yml` + `@vercel/ncc` 번들) | `dist/index.js` 단일 파일로 번들되고, 워크플로에서 `uses:` 로 호출 가능 | E7 |
| E11 | PR 코멘트 게시 (`@actions/github`, 기존 코멘트 **업데이트** 방식) | 같은 PR에 3번 push 해도 코멘트가 1개만 유지되고 내용이 갱신됨 | E10, **예은 Y5** |
| E12 | 머지 차단 — Check Run을 `failure` 로 등록 | 예산 초과 PR에서 GitHub UI에 빨간 ❌ 체크가 뜨고, branch protection 설정 시 머지 버튼 비활성화 | E10, E9 |
| E13 | Action 권한/토큰 문서화 (`permissions: pull-requests: write`) | README 그대로 복붙해서 처음 보는 레포에 붙였을 때 동작 | E11, E12 |
| E14 | 데모용 시연 레포 (`aprice-demo`) 준비 | 통과 PR 1개 / 초과 PR 1개가 미리 열려 있고, 재실행만 하면 되는 상태 | E12, **가영 G7**, **예은 Y6** |

> 💬 **작업량 메모 (은지)**: GitHub Action 인증·권한·Check Run은 이 프로젝트에서 **기술 난이도가 가장 높고 디버깅이 가장 오래 걸리는 구간**이라 은지에게 몰아두었다.
> 대신 은지는 **AST 로직도, HTML/CSS도 건드리지 않는다.** E1(초기 세팅)이 끝나면 Day 2부터는 GitHub 연동에만 집중한다.

**책임 영역**: AST 파싱, 유료 API 호출 지점 탐지, 호출 단위(unit) 추정, 트래픽 시나리오 엔진, 월 비용 계산, 신뢰도(confidence) 산정

## 5. 통합 지점 (Integration Points)

> 여기가 깨지면 **3명의 작업이 동시에 멈춘다.** 각 지점은 정해진 시각에 셋이 모여서 확인한다.

### 🔴 IP-1 · `AnalysisResult` JSON 스키마 확정 — **Day 1 저녁**

가장 중요한 계약. 이게 확정돼야 셋이 **서로를 기다리지 않고** 병렬로 일할 수 있다.

- **참여**: 은지(작성) · 가영(필드 정의) · 예은(표시 요구사항)
- **산출물**: `schemas/analysis-result.schema.json` + `fixtures/sample-result.json`
- **완료 조건**: 3명 모두 승인 + ajv 검증 통과 + main 브랜치에 머지

```jsonc
// fixtures/sample-result.json (구조 예시 — 실제 필드는 Day 1에 확정)
{
  "version": "1",
  "generatedAt": "2026-07-22T09:00:00Z",
  "repo": { "name": "acme/web", "baseRef": "main", "headRef": "feat/chat" },
  "budget": { "monthlyUsd": 300, "currency": "USD" },
  "summary": {
    "totalMonthlyUsd": 412.50,
    "deltaMonthlyUsd": 180.00,   // 이 PR 때문에 늘어난 비용
    "verdict": "fail",            // pass | warn | fail
    "detectionCount": 7
  },
  "traffic": { "dau": 1000, "requestsPerUserPerDay": 3 },
  "detections": [
    {
      "id": "d1",
      "provider": "openai",
      "product": "gpt-4o",
      "file": "src/chat/service.ts",
      "line": 42,
      "snippet": "openai.chat.completions.create({...})",
      "multiplier": 10,
      "multiplierReason": "for 루프 내부 호출 (추정 반복 10회)",
      "estimatedCallsPerMonth": 900000,
      "estimatedMonthlyUsd": 380.00,
      "confidence": "medium",
      "isNew": true
    }
  ],
  "skipped": [
    { "file": "src/legacy/old.js", "reason": "parse_error" }
  ]
}
```

1. **은지 → 예은 이관: Y11 npm 배포**

### STEP 0 — 문제 제기 (30초, 슬라이드)

> "팀에서 GPT 기능 하나 넣었는데, 다음 달 청구서가 400달러였습니다.

- APrIce의 주장: **머지되기 전에, 코드에서, 미리.**

---

### STEP 1 — 개발자의 로컬 실행 (90초, 터미널) ⭐ **메인 경로**

```bash
$ npx aprice scan --base main
```

**보여줄 것:**

1. 스캔 진행 표시 → 결과 표 출력
   ```
   ⚠️  APrIce — 이 변경으로 월 예상 비용이 증가합니다

     provider    product   위치                      월 예상
     openai      gpt-4o    src/chat/service.ts:42    $380.00  ← 루프 내부 호출 ×10
     twilio      sms       src/alert/notify.ts:18    $32.50

     현재 월 예상 총액   $412.50
     설정된 월 예산      $300.00
     초과                $112.50 (137%)

   ✗ 예산을 초과했습니다.
   ```
2. **강조 포인트**: `src/chat/service.ts:42` — for 루프 안의 호출을 잡아내 `×10`을 곱했다는 점
3. 종료 코드 확인
   ```bash
   $ echo $?
   1
   ```
   → "**pre-commit hook에 걸면 커밋 자체가 막힙니다.**"

---

### STEP 2 — 안전망으로서의 GitHub Action (90초, 브라우저)

1. 데모 레포의 PR 화면을 연다 → **APrIce 봇 코멘트**가 이미 달려 있음
2. 코멘트 접힌 상태: 결론 5줄 (총액 / 예산 / 초과분 / 판정)
3. `<details>` 펼치기 → 파일·라인별 상세 표, 각 줄이 **코드 라인으로 링크**
4. 아래로 스크롤 → **Checks 영역의 빨간 ❌ `APrIce / budget-check`**
5. **머지 버튼이 비활성화되어 있는 것을 보여준다** ← 심사위원이 가장 반응하는 지점

> 멘트: "로컬에서 안 돌리고 올린 사람이 있어도, 여기서 한 번 더 막힙니다."

---

### STEP 3 — 비개발자를 위한 정적 리포트 (60초, 브라우저)

1. Action 아티팩트에서 `aprice-report.html` 다운로드 → 더블클릭
2. 보여줄 것:
   - 예산 대비 게이지 (빨강, 137%)
   - provider별 비용 막대
   - 탐지 상세 표 + 신뢰도 배지
3. **멘트**: "서버도 로그인도 없습니다. HTML 파일 하나라서 슬랙에 그냥 올리면 됩니다. 기획자도 열어볼 수 있어요."

---

### STEP 4 — 해결하고 통과시키기 (45초) ⭐ **클라이맥스**

1. 문제의 코드를 고친다 (루프 안 호출 → 배치 호출 1회로 변경, 또는 `gpt-4o` → `gpt-4o-mini`)
2. 재실행
   ```bash
   $ npx aprice scan --base main
   ✓ 월 예상 총액 $71.20 / 예산 $300.00 — 통과
   ```
3. push → PR 화면 새로고침 → **코멘트가 초록으로 갱신되고, 체크가 ✅로 바뀌고, 머지 버튼이 살아난다**

> 멘트: "새 코멘트가 쌓이는 게 아니라 **기존 코멘트가 갱신**됩니다. PR이 지저분해지지 않게요."

## 부록 A. 예상 질문 & 답변

| 질문 | 답변 |
|---|---|
| 정적 분석으로 호출 횟수를 어떻게 아나요? | 정확히는 모릅니다. 그래서 `.aprice.yml` 에 **트래픽 프로필(DAU, 요청/유저/일)** 을 입력받고, 코드에서는 **루프·조건 구조로 배수(multiplier)** 만 추정합니다. 불확실한 건 `confidence: low` 로 표시합니다. |
| 오탐이 나면요? | `// aprice-ignore` 주석과 `.aprice.yml` 의 `exclude` 로 제외할 수 있습니다. 그리고 **경고이지, 자동 삭제가 아닙니다.** |
| 가격이 바뀌면요? | 가격 DB가 레포에 YAML로 있어서 **PR로 누구나 갱신**할 수 있습니다. 각 항목에 `checked_at` 이 있어서 오래된 데이터는 리포트에 경고가 뜹니다. |
| 기존 비용 모니터링 도구와 뭐가 다른가요? | 그건 **사후 관측(observability)**, 이건 **사전 차단(prevention)** 입니다. 돈이 나가기 전에 PR에서 막습니다. |
| 서버 없이 어떻게 팀이 공유하나요? | 공유 대상은 PR 코멘트와 HTML 파일입니다. 서버가 없다는 게 **도입 장벽이 0**이라는 뜻이기도 합니다. |
